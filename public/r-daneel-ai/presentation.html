Part 1 — The Problem

What Are LLM Guardrails?

Mechanisms that constrain, validate, or correct AI output before it reaches the real world

Common types

🚫 Content Filters

Block categories of output — NSFW, PII, competitor names

📐 Output Schemas

Enforce JSON structure, valid enums, required fields

🔍 Retrieval Grounding

Only answer from verified documents (RAG)

⚡ Symbolic Validators

Formal logic checks (conceptually like Design Rule Checking for outputs) — our focus today

The root cause

LLMs are probabilistic — they predict plausible tokens, not verified truths.

# An LLM will confidently…
→ Invoice a customer -$400
→ Book a meeting on Feb 30th
→ Insert string into integer database column
→ Call API with missing required fields
Guardrails are anything sitting between the model's output and the real world. Content filters are the most common. Symbolic validators are different — they use formal logic to check correctness against explicit rules. We focus on symbolic validation in this case study.
Part 1 — The Problem

The Hallucination Gap

LLMs have great intuition and terrible eyesight

✅ LLMs are strong at

  • 🧠 Strategy & high-level planning
  • 💬 Natural language explanation
  • 📚 Domain knowledge & heuristics
  • 🔗 Connecting disparate concepts

❌ LLMs are weak at

  • 📐 Spatial geometry, routing, & keep-out zones
  • 📋 Exact rule compliance
  • 🔢 Precise arithmetic
  • 🔁 Consistent state tracking over time
The gap between "knows the rules" and "reliably follows them" is exactly where symbolic guardrails live.
LLMs absorbed millions of chess games and thousands of pages of theory. They can write beautiful prose about the Queen's Gambit. But ask them to track a board state over 20 moves and they will start proposing invalid coordinates. Consistent spatial state tracking is a fundamental limitation of pure transformer models.
Part 1 — The Problem

Two Guardrail Philosophies

Comparing the error recovery rates of reject-only and explanatory feedback loops

🚫 Blocking (reject-only)

"HTTP 400: Bad Request."

The LLM knows it failed but not why. It retries blindly, loops, guesses fields, wastes tokens, and eventually deadlocks.

Raw baseline: High failure rates. Frequent execution timeouts.

💡 Explaining (structured feedback)

"No — and here is exactly why."

"Cannot place cell. Coordinates [X:12-16, Y:4] overlap with net layout keep-out zone (Cell_B reserved boundary)."

With Explanatory Guardrail: Immediate self-correction. Infinite loops eliminated.
"Our goal is not to control the LLM, but to show direction when it is confused." — R_Daneel_AI design principle
This is the core thesis. A guardrail that only says "no" forces random retries. A guardrail that explains *why* the placement is invalid (e.g. naming the overlapping net or boundary block) gives the LLM something it can reason with on the next attempt. This structured feedback is what eliminates retry loops.
Part 2 — The Tool: Prolog

What Is Prolog?

A logic programming language from 1972 — you declare what is true, not how to compute it

% ── Facts: things that are true ──
parent(tom, bob).
parent(bob, ann).

% ── Rule: what makes something true ──
grandparent(X, Z) :-
    parent(X, Y),
    parent(Y, Z).

% ── Queries: ask the engine ──
?- grandparent(tom, ann).
% → true ✓

?- grandparent(tom, Who).
% → Who = ann ✓

?- grandparent(Who, ann).
% → Who = tom ✓  (works both ways!)

How it works

① Assert facts

State what is true about your domain as simple statements

② Define rules

Write logical relationships using "if → then" Horn clauses

③ Ask queries

The engine proves or disproves, and finds all solutions via automatic backtracking

No loops. No algorithms. No "how". Just "what."
You don't code a loop or function. You say "a grandparent X of Z is any X who is a parent of some Y who is a parent of Z." Then you ask questions. The engine figures out the rest. Fundamentally different from Python or Java.
Part 2 — The Tool: Prolog

Why Prolog Is Perfect for Guardrails

A comparison of Prolog's properties against transformer limitations

🎯 Deterministic

Same query → same answer. Identical inputs yield identical proof paths, eliminating variance.

🔄 Exhaustive by design

Backtracking automatically finds all solutions — perfect for verifying grid alignments and multi-dimensional layout boundaries.

📖 Natively explainable

The proof chain is the explanation. The logic resolution path maps directly to human-readable text.

📝 Declarative & readable

Domain and business rules written in Prolog read almost like official specifications, simplifying system maintenance.

Declarative logic systems provide the formal guarantees that probabilistic systems lack.
These four properties directly address the native weaknesses of LLM agents. LLMs are probabilistic, cannot enumerate exhaustively, and struggle with multi-step constraints. Prolog is deterministic, exhaustive, and natively explainable. This results in a highly complementary system configuration.

Part 3 — Chess & Queen Engine

Introducing Queen 👑

A symbolic chess referee and tactical analyzer built in SWI-Prolog

Queen is not an engine that plays chess.
It's a referee that understands chess — and can explain every ruling.

✅ Move Legality + Explanation

Validates any move in standard coordinates. Returns a structured plain-English explanation if illegal — not just a boolean.

📌 Pin Detection

Identifies "pinned" pieces (pieces blocking an attack on the King that are illegal to move) and explains the check block.

⚔️ Fork & Discovered Attack

Finds "forks" (one piece attacking two targets) and "discovered attacks" (moving a blocking piece to open a slide line of fire).

♟️ Full Tactical Summary

Aggregates checks, pins, threats, forks, material balance into one structured JSON payload the LLM can read directly.

The key distinction: Stockfish is a probabilistic search solver (tells you the BEST move). Queen is a design rule checker (tells the LLM what is LEGAL and WHY). Analogy for physical design: validate_move is a DRC check, pin detection is path blockage, and threats are layout routing conflicts.
Part 3 — Chess & Queen Engine

Queen's Three-Layer Architecture

Each layer has exactly one job — cleanly separable and testable

① Constraint Logic Layer

queen.pl

Pure SWI-Prolog. Chess rules as logical facts: move legality, path blockers, check/mate, pins, forks, discovered attacks. No Python, no I/O.

② Orchestration Layer

queen.py

Python bridge via pyswip. Parses FEN notation into Prolog variables, runs queries, returns structured Python dicts. Orchestrated as a local module.

③ Interface Layer

queen_mcp.py

FastMCP server. Exposes Queen as callable tools over the Model Context Protocol — any compatible LLM host (Claude, Gemini…) can call Queen like a function.

💡 MCP = Model Context Protocol — a universal "USB standard" for LLM tools. Write the tool once, use it with any compatible host.

The Prolog file is the brain — pure logic. Python is the glue — speaks both languages. MCP is the USB socket — plug-and-play with any LLM. Each has one job. Swap any layer independently.

Part 3 — Chess & Queen Engine

The System Prompt: Core Philosophy

Enforcing the boundary between probabilistic planning and symbolic ground truth

# System Prompt — Role Definition
You are a blind grandmaster chess player. Because you are blind, you CANNOT see the board directly, and you have zero spatial vision.
Your assistant, a symbolic Prolog referee, is watching the board for you. The referee's tool outputs are the absolute, non-negotiable ground truth.

To play successfully, you must rely entirely on the referee to describe the board layout, threats, and checks. Any attempt to guess coordinates will result in a blunder. Trust the referee's reports implicitly!

Why the Persona Works

1. Pruning Spatial Hallucinations: Telling the model it is "blind" actively suppresses its reliance on faulty internal spatial representations.

2. Enforced Tool Dependency: The prompt positions the Prolog referee tools as the sole information channel, increasing tool-calling fidelity.

3. Critical Checklist: The prompt mandates verifying check status, double-check scenarios, piece development, and board tension.

By instructing the model that it is "blind" and has zero spatial vision, we suppress the transformer's tendency to predict likely-looking coordinate strings and force it to rely purely on the structured facts injected by Prolog.

Move latency ranges from 24.3s to 276.9s, making the hybrid architecture completely unusable for blitz or rapid chess formats.

📊 Statistical Limits

Evaluations represent single-run statistics on the database. We did not run trials across varying random seed ranges or temperature thresholds.

📉 High-Tier Degradation

At the 2500 Elo tier, hybrids underperform baselines because the planner confidently steers search spaces into complex areas neither understands.

Appendix — Prompt Design

The System Prompt: Minimax Utility

Formulating candidate move selection as a constraint optimization problem

# Mathematical Move Selection Instructions
For every move, you must select 2 to 4 candidate moves and calculate their Net Utility using this formula:

Net Utility = Gain(Me) - MaxGain(Opponent Best Response)

# Tension Capture Penalty:
If a candidate move is a capture that resolves active board tension for a net utility of 0, apply a -0.5 penalty. This ranks it below quiet developing or reinforcing moves (utility 0).

Decision Criteria

1. Mandatory Quiet Moves: At least one candidate move evaluated must be a non-capturing quiet developing move to prevent over-aggression.

2. Tension Management: Capturing immediately is penalized. The prompt forces the LLM to maintain tension and develop.

3. Stubborn Defense: If all candidate utilities are negative, the LLM is instructed to minimize loss, block mates, and complicate the board.

The prompt explicitly defines the minimax utility calculation. This forces the LLM to run simulate_move and get_tactical_summary on the resulting state to identify the opponent's maximum response before making a move, preventing simple tactical oversights.
Appendix — Self-Correction

Prompt Engineering: Self-Correction

Recovering from logical constraint violations via explanation feedback loops

# User Prompt on Validation Failure
Your proposed move "f3e5" was rejected by the referee.

# Validation Error:
"ILLEGAL: Knight on f3 is pinned to the King by the Bishop on b7. Moving it exposes your King to check."

# Instructions:
1. Analyze the check path and pin block.
2. Propose a new move that resolves the constraint violation.
3. Do not propose "f3e5" again.

Why Self-Correction Succeeds

1. Zero Hallucination of Cause: The LLM does not need to guess why it failed. The error message is mathematically exact.

2. Immediate Path Pruning: The rejected move is added to the "dead paths" set in memory, preventing repeated blunders.

3. Semantic Explanation: Prolog's native proof tree translates binary logic failures into semantic context the transformer excels at processing.

If the LLM proposes an illegal move, the Prolog referee returns the exact reason (e.g., piece pinned to King). This error is fed back to the LLM, prompting it to reflect on the blunder and choose a valid path.
Appendix — Prompt Optimization

Automated Prompt Optimization

Proposed loop for systematically evolving agent instructions from execution logs

# Planned Meta-Optimization Loop
For each puzzle/game execution:
  Parse thoughts and Prolog rejections in log files.
  Identify cognitive mismatches (e.g. LLM ignored check constraint).

# Planned Prompts Evolution Cycle:
If rejections for PINNED_PIECE > threshold:
  Synthesize explicit prompt rule: "Verify pinned_pieces list before proposing."
  Run 500-puzzle benchmark suite to measure Elo delta.
  Commit prompt update if Elo delta is positive and no regression.

Planned Optimization Pipeline

1. Autopsy log analysis: A higher-tier model (e.g., Fable 5) will evaluate execution trace CSVs/JSONLs of lower-tier agents (e.g., gemma-2-27b).

2. Systematic Rule Synthesis: Instead of manual trial-and-error, failure traces will be mapped to declarative text prompts (such as adding the $-0.5$ tension penalty).

3. Continuous Evaluation: The optimization loop will utilize Lichess puzzle suites as an automated regression testbed to validate prompt edits.

4. Prompt Compilers: Evolving to dynamic prompt compilers (such as DSPy) to generate specialized constraint weights dynamically per board-state profile.
